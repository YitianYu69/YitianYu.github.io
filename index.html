Efficient Training and Activation Compression

One of my major technical interests is making deep learning systems more efficient without losing their effectiveness. In this line of work, I have explored activation compression, hardware-aware training, and memory-efficient techniques for modern vision models. My goal has been to understand how to reduce training cost while still preserving useful learning dynamics. This kind of project requires both algorithmic thinking and systems awareness, since the best solution is rarely only about accuracy or only about speed. It is about the tradeoff between the two.

Through this project, I have worked with PyTorch-based training pipelines, custom optimization experiments, and implementation details related to large-scale model training. I am especially interested in how ideas such as lower-bit representations, better initialization, and structured training strategies interact with convergence and resource usage. The broader value of this work is that it connects theory with real engineering constraints. For future employers and collaborators, it demonstrates my ability to take a complex systems problem, analyze bottlenecks, and build toward a more practical solution.

Discovery Project: Activation Compression Trainer

For my ECE 1100 Discovery Project, I decided to build an activation compression trainer. I picked this project because it matched what I had actually been spending time on already: efficient deep learning training, memory reduction, and GPU-aware implementation. The main idea was to reduce the memory used to store activations during backpropagation, since that becomes a real bottleneck in training.

The main result I got was that I could apply uniform 2-bit activation compression across all layers. Compared to FP32 activation storage, that takes each stored activation from 32 bits down to 2 bits, which is 16× smaller and gives 93.75% raw activation storage savings. Compared to FP16 or BF16 activations, it is 8× smaller, which corresponds to 87.5% raw activation storage savings. These numbers are specifically about activation storage, which was the part of training memory I was targeting.

PyTorch Triton CUDA 2-Bit Activations Memory Efficiency ML Systems

The Problem

In standard neural network training, activations have to be saved for the backward pass, and that can use a lot of GPU memory. The larger the model or batch size gets, the more painful that storage cost becomes. I wanted to work on that bottleneck directly instead of only treating memory use as a side effect of training.

The Process and Progress

I started by looking more carefully at where activation memory was being used and what part of the training pipeline I could realistically change. Over the semester, I built the trainer, tested different implementation ideas, debugged memory and performance issues, and kept refining the design until I could support uniform 2-bit activation compression across all layers. That process taught me a lot about how high-level training code connects to lower-level GPU behavior.

ECE Skills Gained

The main ECE skill I used in this project came from ECE 3803: General GPU Programming. The CUDA ideas from that class made the project much more concrete for me, because I was no longer thinking only about models. I was also thinking about parallel execution, memory movement, and performance. I used PyTorch for the training framework and Triton for GPU-efficient implementation ideas.

Successes, Challenges, and Final Thoughts

The biggest success for me was that this was a real systems project, not just a model that I trained once. I built something tied to an actual bottleneck, and I got a clear result from it. The hardest part was debugging compression and memory-related behavior while still keeping the implementation practical. Overall, this project made me even more interested in deep learning systems, efficient training, and GPU-aware machine learning work.
